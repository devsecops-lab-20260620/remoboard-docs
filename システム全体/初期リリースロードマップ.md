# 初期リリースロードマップ

## 目的

- 遠隔お知らせ掲示板システムの初期リリースに向けた、段階的な開発順序を明確にする
- どのリポジトリを、どの順番で、どの水準まで作るかを整理する
- 初期リリースで必要な機能と、後回しにできる機能を切り分ける

## 参照元

- [システム初期検討.md](./システム初期検討.md)
- [リポジトリ構成と責務.md](./リポジトリ構成と責務.md)

## ロードマップの考え方

- 初期リリースは、最低限の機能から順に積み上げる
- 各マイルストーンは、対象リポジトリの作成と最低限の動作確認を完了条件とする
- マイルストーン4で「基本の機能」として初期リリース成立を狙い、マイルストーン5で拡張機能を追加する

## マイルストーン一覧

### マイルストーン1: `api-admin`, `web-admin` の作成

- 対象リポジトリ
  - `remoboard-cloud-api-admin`
  - `remoboard-cloud-web-admin`
- 目的
  - 管理者向けの基盤を先に作る
  - ユーザー管理、ゲートウェイ管理、監査確認の入り口を用意する
- 主な作業
  - 管理者ログインの土台作成
  - ユーザー一覧、ゲートウェイ一覧、監査ログの表示基盤作成
  - 管理系 API のルーティング、認証、権限制御の骨格作成
  - GitHub Actions の静的解析と単体テストの土台作成
  - Dockerfile の整備とコンテナイメージビルド確認
- 完了条件
  - 管理者向けの画面と API が起動できる
  - 最低限のモックまたはダミーデータで画面遷移を確認できる
  - CI で静的解析と単体テストが実行できる

### マイルストーン2: `api-user`, `web-user` の作成

- 対象リポジトリ
  - `remoboard-cloud-api-user`
  - `remoboard-cloud-web-user`
- 目的
  - 一般ユーザー向けの基本機能を作る
  - ログイン、端末管理、お知らせ管理、配信結果確認の入口を用意する
- 主な作業
  - 一般ユーザー向けログインの土台作成
  - ゲートウェイ一覧、端末一覧、お知らせ一覧の表示基盤作成
  - 端末登録、削除、紐づけの API 骨格作成
  - Python / Node の静的解析と単体テストを各リポジトリで整備
  - PR 時に CI が通る状態へ拡張
- 完了条件
  - 一般ユーザー向けの画面と API が起動できる
  - 最低限のモックまたはダミーデータで基本操作を確認できる
  - CI で各言語の静的解析と単体テストが実行できる

### マイルストーン3: `delivery`, `gateway-api` の作成

- 対象リポジトリ
  - `remoboard-cloud-delivery`
  - `remoboard-gateway-api`
- 目的
  - 配信経路をつなぎ、クラウドから自宅端末までの流れを確認する
  - 端末一覧返却、クラウドからの指示受信、端末表示反映を確認できるようにする
- 主な作業
  - 配信ジョブ投入処理の作成
  - 配信対象端末の解決処理の作成
  - ゲートウェイへの配信要求処理の作成
  - 模擬ゲートウェイの起動処理、受信処理、表示反映確認処理の作成
  - コンテナイメージのビルド設定を整備
  - コンテナイメージの push 手順を CI に追加
- 完了条件
  - クラウド側からゲートウェイ側への配信経路が通る
  - 最低限の配信結果反映を確認できる
  - コンテナイメージのビルドと push が自動化されている

### マイルストーン4: 基本の機能で更新

- 目的
  - マイルストーン1〜3で作成した機能を初期リリース水準まで強化する
  - 実運用を意識した画面、API、配信、状態確認の整備を行う
- 主な作業
  - ログイン後の画面遷移や一覧表示の改善
  - 端末追加・削除・紐づけの操作性改善
  - 配信結果確認や状態確認の表示改善
  - 管理者向けのユーザー管理・ゲートウェイ管理の改善
  - CI の静的解析、単体テスト、コンテナビルド、push を全リポジトリで安定稼働させる
- 完了条件
  - 初期リリースとして最低限の利用フローが成立する
  - 必須機能の操作が一通り確認できる
  - CI が全リポジトリで一貫したルールで動作する

### マイルストーン5: 最終の機能で更新

- 目的
  - 初期リリース後の拡張機能や運用改善を追加する
  - 後回しにした機能を完成させる
- 主な作業
  - お知らせの予約配信
  - 配信失敗時の高度な再送制御
  - 契約プランや利用量課金の詳細管理
  - 高度な分析画面や統計ダッシュボード
  - 障害通知の自動化や外部通知連携
  - CI / CD のタグ運用やリリース運用を最終整備
- 完了条件
  - 初期リリース後の拡張機能として追加導入できる
  - 運用改善を含む高度な機能が利用できる
  - CI / CD の運用ルールが定着している

## 初期リリースで優先する範囲

- ユーザーのログイン
- ゲートウェイ一覧・端末一覧の表示
- 端末の追加、削除、紐づけ
- お知らせの即時配信
- 配信結果の確認
- 管理者によるユーザー管理、ゲートウェイ管理
- ゲートウェイおよび端末の状態確認

## 初期リリースでは後回しにできる範囲

- お知らせの予約配信
- 配信失敗時の高度な再送制御
- 契約プランや利用量課金の詳細管理
- 高度な分析画面や統計ダッシュボード
- 障害通知の自動化や外部通知連携

## 受け入れ基準

- マイルストーン4までで、初期リリースに必要な基本フローが完了していること
- 画面、API、配信処理、ゲートウェイ連携の最低限の流れが確認できること
- 後回し機能が未実装でも、初期リリースの成立に支障がないこと
